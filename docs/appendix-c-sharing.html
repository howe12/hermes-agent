§ 1

问题定义 · "共享"指什么

你问的"共享"是个有歧义的词。Hermes 的设计里至少有 4 种"共享"，目的和实现完全不同：

  • 同一台机器，两个 hermes 安装——比如你想 NUC 和 Spark 机器人各装一个，但想共享 skill 库。
  • 同一台机器，多个 hermes 实例——比如一个装来写代码，一个装来控制机器人，要互相隔离。
  • 不同机器，搬家——比如在 NUC 上跑了 1 周，要把全部状态搬到你的笔记本。
  • 不同机器，实时共享——比如 NUC 的 hermes 和你的笔记本的 hermes 看到的学生画像是同步的。

每种共享对应 不同的代码机制。混淆就会出错——比如你不会用"zip 搬家"去做"实时同步"（性能/一致性都不行），也不会用"实时共享"去做"完全隔离"（隐私不行）。

先说一个全局原则：Hermes 没有任何机制支持"多 hermes 同时写同一个数据"——那会破坏 SQLite 的 ACID 假设。所以下面所有"共享"方案，要么是单向读（备份还原、profile 克隆），要么是后端代为同步（honcho）。这个原则决定了 4 种机制各自的适用场景。
§ 2

4 种机制对照矩阵

L1 · 文件系统层
共享 HERMES_HOME
两个 hermes 进程读同一个 ~/.hermes/ 目录。用 NFS / SMB / iCloud / Dropbox / 软链。
适合：单机多用户、NUC + 笔记本同账号、文件级协作
L2 · 同一机器多实例
hermes profile
hermes profile create/use 在同一安装下切换多个隔离实例。源 = hermes_cli/subcommands/profile.py。
适合：同一台机器、多个不同任务场景、配置完全隔离
L3 · 跨机器搬家
hermes backup + import
hermes backup 打包整个 ~/.hermes/ 为 zip，hermes import 还原。源 = hermes_cli/backup.py（1064 行）。
适合：换机器、迁移、灾备、定时快照
L4 · 跨设备实时共享
共享 honcho 后端
切换 memory provider 到 honcho，多 hermes 实例通过同一 workspace 共享 user model。源 = plugins/memory/honcho/。
适合：多端协同（手机+笔记本+NUC）、跨设备学情画像
L1 共享目录 L2 profile L3 backup+import L4 honcho
跨机器？ ✓（NFS/SMB） ✗ 本机 ✓（zip 拷） ✓
实时同步？ 几乎实时（看文件 sync 延迟） N/A 手动（每次 backup/import） ✓ 实时
支持写冲突？ ✗（SQLite 强一致性假设） N/A N/A（一次性） ✓ honcho 协调
需要网络/服务？ 本地网络 无 无 需要 honcho-ai 服务
隔离度 完全共享 完全隔离 完全独立 共享 user model，独立 session
配置难度 低 低 极低 中（要注册 honcho-ai）
一句话：前 3 个都是本地 + 文件的方案，第 4 个才是真正的分布式 + 实时。根据你的问题形态选一个，不是 4 个都上。
§ 3

L1 · 共享 HERMES_HOME 目录（最朴素）

所有 hermes 数据都在 ~/.hermes/（或 $HERMES_HOME）。最直接的"共享"就是让两个 hermes 读同一个目录。

实现方式

  • 符号链接：在第二台机器 ln -s /mnt/shared/leo/.hermes ~/.hermes
  • NFS / SMB 挂载：把 NUC 的 ~/.hermes/ 挂到笔记本
  • Dropbox / iCloud / Syncthing：直接把目录放云同步盘里（⚠️ 见下面警告）

机制

Hermes 的所有数据访问都走 hermes_constants.get_hermes_home()：

# hermes_constants.py 真实代码（行 53-65）
def get_hermes_home() -> Path:
    # 1. 读 HERMES_HOME 环境变量
    # 2. 否则读 active_profile 配置文件
    # 3. 否则用平台默认 (~/.hermes/)
    return Path(os.environ.get("HERMES_HOME") or _default_path())

所以你可以：

# 在第二台机器（笔记本）
export HERMES_HOME=/mnt/nuc-leo/.hermes
hermes chat

适用场景

  • 同账号在 NUC + 笔记本 + 手机上跑 hermes
  • 家庭/小团队多设备同步（家里一台 NUC 常开）
  • 不需要"实时"——下班前 NUC 跑完，明天打开笔记本看到
⚠️ SQLite + 文件同步盘的硬伤：Hermes 用的 SQLite 假设同一台机器单进程写。如果你同时在两台机器跑 hermes，SQLite 必崩。所以 L1 真正安全的使用方式是：同一时间只在一台机器上跑 hermes，另一台只是"读 + 偶尔重启接管"。如果你需要"两台同时跑"——必须升级到 L4 honcho。
§ 4

L2 · hermes profile（同一台机器多实例）

文件 hermes_cli/subcommands/profile.py（203 行）。

是什么

hermes profile 在同一台机器同一份 hermes 安装下，开多个隔离的实例。每个实例有自己的 ~/.hermes/profiles/<name>/ 目录。

关键命令

# 真实 subcommand（profile.py 里的 build_profile_parser）
hermes profile list                              # 列出所有 profile
hermes profile use <name>                        # 切到某 profile（sticky）
hermes profile create <name> [--clone] [--clone-all] [--clone-from SRC]
                                                 # 新建（可选从已有克隆）
hermes profile delete <name> -y                  # 删除
hermes profile describe <name> [--auto]         # 设置/读描述（给 kanban 用）

2 种克隆模式

  • --clone：复制 config / .env / SOUL.md / skills（启动用的）
  • --clone-all：全量复制（包括所有 state）

适合什么场景

profile: code · profile: robot · profile: tutor
  • profile: code —— 用 Claude Sonnet 写代码，关闭机器人 skill
  • profile: robot —— 用 DeepSeek 跑机械臂，全开机器人 skill
  • profile: tutor —— 教育场景，对接学生学情

这三个 profile 互不干扰——你切换 profile 时 LLM 配置、skill 集合、memory 都跟着切。

关键设计点：profile 不是为了"共享"，是为了"隔离"。如果你想"同一台机器装两个 hermes"——profile 就是官方答案。不要手动建两个 ~/.hermes/ 目录，profile 系统已经管了。
§ 5

L3 · hermes backup + import（zip 整包搬家）

文件 hermes_cli/backup.py（1064 行）。

两条命令

# 1. 备份（hermes_cli/main.py:4231）
hermes backup                              # 整包打 zip → ~/hermes-backup-20260613.zip
hermes backup -o /tmp/snap.zip             # 指定输出
hermes backup --quick                      # 只备份关键状态（快）
hermes backup --quick -l daily             # + label

# 2. 还原（main.py:4243）
hermes import /tmp/snap.zip                # 还原
hermes import /tmp/snap.zip -f             # 覆盖不提示

backup 包含什么

从 _EXCLUDED_DIRS、_EXCLUDED_SUFFIXES、_EXCLUDED_NAMES 看：

# hermes_cli/backup.py:28-66 真实代码
_EXCLUDED_DIRS = {
    "hermes-agent",    # 仓库本身（重新 clone）
    "__pycache__",     # 字节码缓存
    ".git",            # git 目录
    "node_modules",    # js 依赖
    "backups",         # 嵌套备份（防指数爆炸）
    "checkpoints",     # 会话本地轨迹缓存（不可移植）
}

_EXCLUDED_SUFFIXES = (
    ".pyc", ".pyo",
    # SQLite sidecar：WAL/SHM/journal 跳过，
    # 避免快照+旧 sidecar 拼出撕裂状态
    ".db-wal", ".db-shm", ".db-journal",
)

_EXCLUDED_NAMES = {
    "gateway.pid",   # 运行时 pid（换机器无意义）
    "cron.pid",
}

所以 backup 实际打包的是：所有 state + 全部 skill + cron 配置 + auth + config.yaml + .env + sessions.db。

--quick 模式

只备份关键状态文件，适合每天自动备份：

  • config.yaml
  • state.db（这是最关键的，所有 session + FTS5 索引）
  • .env（API keys）
  • auth.json（OAuth 凭证）
  • cron/

不含 skills、不含 detailed sessions。恢复时 hermes 重新下载 skills 就行。

适用场景

NUC → zip → scp → 笔记本 → hermes import 新机器满血
  • 换机器（笔记本换台式机）
  • 团队/家庭内部传递（"这是我配好的 hermes"）
  • 灾备（每天 cron 跑 hermes backup --quick）
  • 实验 / 试错（备份 → 改坏 → import 还原）
SQLite 一致性保护：备份时 Hermes 调 sqlite3.backup()（行 ~140）做原子快照，不是粗暴 cp。这就是为什么 .db-wal/.db-shm/.db-journal 都被排除——它们是 SQLite 运行时 sidecar，跟 atomic snapshot 配对会出"撕裂状态"。
§ 5.5

🧬 "继承"模式 · 专治"上一代传记忆给下一代"

如果你的诉求是 "我不需要同时跑两台 hermes，只需要这一代 hermes 退场时把记忆交给下一代 hermes"——你需要的不是"共享"，是"继承"。这正是 L3 backup+import 的天然语义。

一句话总结：hermes backup = 父节点立遗嘱 · hermes import = 子节点接受遗嘱。中间不依赖 honcho、不依赖云服务、不依赖 SQLite 共享。

3 个最常见的"继承"剧本

📕 剧本 1

同一台 NUC 上 · 上一代 profile → 下一代 profile

典型场景：你今天在 NUC 上用 tutor profile 教学了半年，沉淀了大量学生错例、ROS2 教学脚本、机械臂调试记录。下学期你想 完全重来（"tutor-v2"），但 想保留旧的全部记忆作为"前任教学档案"备查。

# 1. 在旧 profile 上立遗嘱
$ hermes profile use tutor
$ hermes backup --quick -o tutor-2026春.zip

# 2. 克隆出一个 v2，import 旧记忆作为"地基"
$ hermes profile create tutor-v2
$ hermes profile use tutor-v2
$ hermes import tutor-2026春.zip --prefix inherited_

关键在 --prefix inherited_——所有继承来的会话自动加前缀，跟 v2 自己的新会话视觉上区分开。这就是"子承父业"：继承 + 独立。

📗 剧本 2

跨设备 · NUC 旧机 → 笔记本新机（一次性迁移）

典型场景：你换了开发机，NUC 上的全部教学档案、机器人调试记录、课程笔记要 一次性 搬到笔记本。

# === 在 NUC（旧机）上 ===
$ hermes backup -o ~/aquarius-psyche-NUC-2026-06-16.zip
$ scp ~/aquarius-psyche-NUC-2026-06-16.zip leo@newlaptop:~/

# === 在笔记本（新机）上 ===
$ pip install hermes-agent            # 先装空壳
$ hermes import ~/aquarius-psyche-NUC-2026-06-16.zip
$ hermes doctor                       # 验证

零配置。NUC 上跑了 3 个月的 hermes，原封不动在笔记本上复活——连 MEMORY.md / USER.md / 长期记忆库 / 技能库全带走。

📘 剧本 3

自动化 · 每周 cron 拍一张"快照"，回滚用

典型场景：你怕某次系统升级 / 误操作把 ~/.hermes/ 搞炸，想 有后悔药。

# ~/.hermes/cron.d/weekly-snapshot.cron
0 3 * * 0 hermes backup --quick -o /mnt/nas/hermes-snapshots/$(date +\%Y\%m\%d).zip

每周日凌晨 3 点，自动 把整包拍到 NAS。出问题就 hermes import /mnt/nas/hermes-snapshots/20260615.zip，回到上周。注意用 --quick 跳过 hermes-agent 源码目录（几百 MB）——只备份"状态 + 记忆 + 技能"。

"继承" vs "共享" 的核心区别

共享（实时）继承（一次性）
代表方案L1 共享目录 / L4 honchoL3 backup+import
时间模型两台同时活一台死、一台生
冲突处理需要（SQLite / honcho）不需要（旧的冻结，新的覆盖）
外部依赖L1 需 NAS · L4 需云零（一个 zip 文件）
你的场景匹配度✗（"不需要同时"）✓ 完美匹配
对你的最优解：你的诉求 = "继承"（不是"共享"） → 锁死 L3 backup+import。剧本 1（同机 profile 轮换）+ 剧本 3（cron 每周快照）组合用——够用一辈子，不需要 L1 共享目录、不需要 L4 honcho。

一些工程上的小心思

  • zip 加密：Hermes 的 backup zip 默认不加密——但 zip 里有你的 .env（含 API key）。传给同事 / 上传云前 务必 手动加密码（zip -e）或先清理
  • 排除大文件：backup 默认排除 hermes-agent/ 源码和 *.log，但不排除 视频/截图等大附件。如果你的 ~/.hermes/ 里有 10G 课程视频，backup 会变胖。可以用 --exclude 模式（具体参数看 hermes backup --help）
  • 版本对齐：备份的 hermes 版本 ≠ 恢复的 hermes 版本时，极个别 字段可能 schema 漂移。保险做法：hermes doctor 在 import 后跑一遍
  • 跨平台：Linux 备份的 zip 在 macOS 上 import 完全 OK（Hermes 跨平台），但 Windows 上 zip 解码有点小坑——建议 NUC → 笔记本都走 Linux
§ 6

L4 · 共享 honcho 后端（云端实时同步）

文件 plugins/memory/honcho/。这是唯一真正"实时多端共享"的方案。

honcho 是什么

Honcho 是 AI-native memory 库，专门为多端场景设计。从 plugins/memory/honcho/plugin.yaml：

description: "Honcho AI-native memory — cross-session user modeling
  with dialectic Q&A, semantic search, and persistent conclusions."

关键短语：cross-session user modeling。它不只存"对话内容"，而是建立用户模型——"leo 在机械臂上踩过坑"、"leo 偏好简洁回答"。

它怎么"共享"

从 plugins/memory/honcho/client.py 头部的解析顺序：

# honcho client 配置解析顺序
1. $HERMES_HOME/honcho.json          # 实例级（隔离用）
2. ~/.honcho/config.json            # 全局（跨 hermes 共享）
3. 环境变量 HONCHO_API_KEY 等       # 临时

关键在 第 2 个——~/.honcho/config.json 是用户级配置，所有 hermes 实例共享。你装在 NUC 和笔记本的两个 hermes，只要都用 ~/.honcho/config.json 里的同一组 credentials，就自动共享 honcho workspace。

共享的是什么

Honcho 内部是 workspace（= 一个共享知识库）+ peer（= 一个用户/AI 角色）：

  • peer card：关于某个用户的"持久结论卡"——"leo 的机器人项目在 ROS2 Humble"
  • semantic search：跨设备的对话历史语义检索（"上次他提过 PID 调参"）
  • dialectic 模式：两台设备上的 hermes 都能"读 + 写"同一张 peer card，Honcho 后端协调

机制示意

NUC hermes → Honcho 云 ← 笔记本 hermes

两台 hermes 都通过 honcho-ai SDK 跟同一个 workspace对话。Honcho 后端处理冲突、合并 user model、保持一致。

profile 隔离 vs honcho 共享

从 client.py 还能看到一个精细设计：

# client.py: profile_host_key()
def profile_host_key(profile):
    if not profile or profile in {"default", "custom"}:
        return "hermes"           # 默认 profile → host="hermes"
    return f"hermes_{profile}"  # 其他 profile → host="hermes_xxx"

这意味着 profile 和 honcho 共享可以同时用：

  • 不同 profile 独立（peer card 互不可见）
  • 同一 profile 的多台 hermes 共享（peer card 同步）

需要什么

  • 注册 honcho.ai 账号
  • 把 HONCHO_API_KEY 写到 ~/.honcho/config.json 或 ~/.hermes/.env
  • Hermes config 加 memory.provider: honcho
对机器人 + 教育场景的意义：这是唯一一个让"NUC 的 hermes 看见学生在笔记本上问的问题"的方案。错例库 / 学情档案 / 用户偏好，所有设备自动一致，学生/老师在哪台设备提问都看到一样的"leo 的学习历史"。
§ 7

怎么选 · 决策流程

用决策树回答你的"什么方式共享"问题：

1. 你要"跨设备"还是"同设备"？

  • 同设备多场景 → L2 profile
  • 跨设备 → 继续问

2. 跨设备要"实时"还是"手动"？

  • 实时同步 → L4 honcho（要云服务）
  • 手动 / 定时 → 继续问

3. 跨设备"手动"场景：是迁移还是共用？

  • 迁移 / 灾备 → L3 backup+import
  • 共用同一份数据 → L1 共享 HERMES_HOME（但不能同时跑）

决策矩阵速查

你的场景 推荐
NUC 常开 + 笔记本偶尔用 + 不同时跑L1 共享目录
同一台机器分 "写代码"/"控制机器人"/"教学"L2 profile
换机器 / 给团队成员预装 hermesL3 backup+import
多设备同时用且要看到一致的学生画像L4 honcho
每天自动备份到云L3 backup + cron
手机 + 笔记本 + NUC 同时给一个学生用L4 honcho（必须）
§ 8

对机器人 + 教育场景的实操

1. 你大概率 1 个 L4 + 1 个 L2 就够了

L4 honcho：NUC（机器人控制）+ 笔记本（备课/调试）+ 飞书客户端（学生/老师提问）共享同一份 user model。学情画像、错例库、用户偏好实时一致。

L2 profile：同一台 NUC 上分 2 个 profile——"tutor"（教学场景，默认开 honcho）和 "dev"（开发场景，关闭 honcho 避免被学生历史污染）。

2. L1 不推荐

在你的 NUC + Spark 机器人（如果真接）之间，SQLite 文件共享几乎肯定踩坑。建议直接跳到 L4。

3. L3 用作"每周快照"

设个 cron 每周日凌晨跑 hermes backup --quick，zip 丢到 ~/hermes-snapshots/，万一配置搞坏了能回滚。

4. 时间线

  • 立即：L2 profile 切分 "tutor" / "dev"
  • 第 1 周末：注册 honcho.ai，配 L4
  • 第 2 周：设 L3 backup cron（每周一次）
  • 不推荐：L1 共享目录（除非你确认不会同时跑）
回到主阅读手册：这个附录回答的"共享"问题，跟之前的"长期记忆"和"技能沉淀"是正交的——L1-L4 是"数据放在哪"，"长期记忆 / 技能沉淀"是"数据怎么被用"。
组合起来：L4 honcho 提供"跨设备学情同步"，长期记忆 FTS5 提供"对话检索"，技能沉淀提供"自动学习"。这三层不冲突，是 互补的。
