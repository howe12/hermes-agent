Hermes 技能沉淀 · 持续学习的真正机制

长期记忆是"存档"，技能沉淀才是"学习"。讲清 Hermes 怎么从对话里提炼出可复用的 skill，以及两个真实触发点。

§ 1

为什么"技能沉淀"才是持续学习

你在 Q1 问"Hermes 的持续学习是不是你想要的"。这个问题其实有两个不同的技术：

L1 · 长期记忆（附录 A）
FTS5 全文检索
所有对话自动进 SQLite 数据库。LLM 调 session_search 就能搜历史。
"存档" — 不主动提炼
L2 · 技能沉淀（附录 B）
SKILL.md 自动写盘
对话结束后后台线程分析，把经验提炼成 ~/.hermes/skills/<name>/SKILL.md。
"学习" — 主动提炼
关键区别：长期记忆是被动的——你搜什么它就给你什么。技能沉淀是主动的——它判断这次对话里有没有值得固化的经验，主动写盘。下次同类问题 LLM 直接调这个 skill，不用重新发明。

所以如果你 Q1 的答案是"我想要持续学习"，那 80% 的价值在技能沉淀，长期记忆只是基础。

§ 2

2 个独立触发点 · 一热一冷

技能沉淀有 2 个触发点，目的完全不同：

触发 1 · 每轮对话后 触发 2 · CLI 启动时
位置 agent/turn_finalizer.py:376-398 cli.py:11039 + agent/curator.py:1817
频率 每轮都可能（高） 默认 7 天 1 次（低）
目的 写新东西（新 skill、patch、support file） 清理旧东西（stale → archive）
机制 fork AIAgent daemon 线程 fork AIAgent daemon 线程
用户感知 有时终端闪一行 💾 ... 摘要 几乎不感知（启动 1-2s 后台跑）
谁能用 所有会话（CLI / 飞书 / Telegram） CLI 启动时（gateway 启动也会调）
注意："频率"和"目的"是两套独立的齿轮。
· 触发 1 高频 → 累库（每轮都可能加东西）
· 触发 2 低频 → 瘦身（每周清理一次）
二者配合，技能库既越来越丰富又不臃肿。
§ 3

触发 1 · 每轮对话后 · 后台线程

文件 agent/turn_finalizer.py:376-398。每轮对话正常结束后，在响应交付给用户之后，会做这个判断：

# turn_finalizer.py 真实代码（行 376-398）
_should_review_skills = False
if (agent._skill_nudge_interval > 0
        and agent._iters_since_skill >= agent._skill_nudge_interval
        and "skill_manage" in agent.valid_tool_names):
    _should_review_skills = True
    agent._iters_since_skill = 0

# 条件满足后起后台线程
if final_response and not interrupted and (_should_review_memory or _should_review_skills):
    try:
        agent._spawn_background_review(
            messages_snapshot=list(messages),
            review_memory=_should_review_memory,
            review_skills=_should_review_skills,
        )
    except Exception:
        pass  # Background review is best-effort

3 个 AND 条件

  1. _skill_nudge_interval > 0 —— 开关开着。默认多少？看下面。
  2. _iters_since_skill >= _skill_nudge_interval —— 本轮用了多少次工具调用超过阈值
  3. "skill_manage" in valid_tool_names —— 当前 session 有 skill 写权限

关键的"它不动用户"

  • 起的是 daemon 线程，主对话不阻塞
  • 如果 interrupted（用户 Ctrl+C），不触发
  • review agent 用 同一个 cached system prompt，命中 LLM 前缀缓存，不污染主会话
  • review agent 只能调 memory 和 skill_manage 两个工具，工具白名单隔离
默认 interval 是多少？我没在源码里找到硬编码默认值（行 372 没有赋值），所以默认是 AIAgent.__init__ 里设的某个值。你可以在 config.yaml 改 skill_nudge_interval。如果你嫌太频繁，设大；嫌学得慢，设小。
§ 4

触发 2 · CLI 启动时 · curator 清理

文件 cli.py:11039 + agent/curator.py:1817。不是每轮，是 CLI 启动时检查一次。

# cli.py:11039 真实代码（CLI 启动时）
try:
    from agent.curator import maybe_run_curator
    maybe_run_curator(
        idle_for_seconds=float("inf"),  # CLI 启动 = 完全 idle
        on_summary=lambda msg: self._console_print(
            f"[dim #6b7684]💾 {msg}[/]"
        ),
    )
except Exception:
    pass
# curator.py:1817 maybe_run_curator 真实代码
def maybe_run_curator(*, idle_for_seconds=None, on_summary=None):
    try:
        if not should_run_now():
            return None
        if idle_for_seconds is not None:
            min_idle_s = get_min_idle_hours() * 3600.0
            if idle_for_seconds < min_idle_s:
                return None
        return run_curator_review(on_summary=on_summary)
    except Exception as e:
        logger.debug(f"maybe_run_curator failed: {e}")
        return None

3 个默认常量（curator.py:56-59）

DEFAULT_INTERVAL_HOURS = 24 * 7    # 168 小时 = 7 天
DEFAULT_MIN_IDLE_HOURS = 2           # 至少 2 小时 idle 才跑
DEFAULT_STALE_AFTER_DAYS = 30        # 30 天没用 → stale
DEFAULT_ARCHIVE_AFTER_DAYS = 90       # 90 天没用 → archive

自动状态转移

active → 30 天没用 → stale → 90 天没用 → archived

关键不变式（curator.py:23-26）

  • 只动 agent-created skill（不碰 bundled/官方）
  • 永不自动删除——只 archive（可恢复）
  • pinned skill 跳过所有自动转移
  • 用辅助 client——绝不污染主会话的 prompt cache
首次安装的友好行为：第一次 CLI 启动时 last_run_at 为空，curator 不会立即跑（避免刚装完就重写你的 skill 库），而是 种下一个时间戳，等满 7 天才真的跑第一次。这避免了新装用户的"被自动改了"的体验。
§ 5

_SKILL_REVIEW_PROMPT 全文与解读

触发 1 启动后台 review agent 时，塞给 review agent 的指令是这个。原文来自 agent/background_review.py:45-110。简化版：

# 这是 review agent 看到的 user message
"Review the conversation above and update the skill library.
Be ACTIVE — most sessions produce at least one skill update, even
if small. A pass that does nothing is a missed learning opportunity,
not a neutral outcome.

Target shape of the library: CLASS-LEVEL skills, each with a rich
SKILL.md and a references/ directory for session-specific detail.
Not a long flat list of narrow one-session-one-skill entries.

Signals to look for (any one of these warrants action):
  • User corrected your style/tone/format/legibility/verbosity.
  • User corrected your workflow/approach/sequence of steps.
  • Non-trivial technique, fix, workaround, debugging path emerged.
  • A skill loaded this session turned out wrong/outdated.

Preference order — prefer the earliest action that fits:
  1. UPDATE a currently-loaded skill (PATCH the one in play)
  2. UPDATE an existing umbrella (class-level) skill
  3. ADD a support file under an existing umbrella
     (references/ for notes, templates/ for starters, scripts/ for runs)
  4. CREATE a new class-level umbrella skill

User-preference embedding: when the user expressed style/format/workflow
preference, the update belongs in the SKILL.md body, not just in memory.
Memory captures 'who the user is'; skills capture 'how to do this
class of task for this user'."

3 个隐含的设计哲学

  1. "Be ACTIVE" —— 明确要求 review agent 不要 没事就退出。"A pass that does nothing is a missed learning opportunity" 这话很严厉，但说明 Hermes 团队在调试中发现了"太被动"的失败模式。
  2. "CLASS-LEVEL, not session-level" —— 命名必须是 class-level（"ros2-bridge"），不能是 session-level（"fix-today-bug"）。第 4 种方式明文说"If the proposed name only makes sense for today's task, it's wrong"。
  3. SKILL.md > memory —— 用户偏好必须写进 SKILL.md，不只是 memory。原因：下次 session 启动时 memory 是注入到 system prompt 的，但 SKILL.md 是按需加载的——偏好必须 按需加载才正确（不同任务场景用不同偏好）。
§ 6

4 种更新方式 · 优先级

review agent 拿到对话后，按这个优先级决定写什么。这是精心设计的——大多数更新走 1-2-3（已有 skill），很少走 4（新建）。

优先级 动作 触发条件 写盘路径
① PATCH 当前已加载的 skill 对话里 /skill-name 调过某个 skill，或 skill_view 读过 ~/.hermes/skills/<name>/SKILL.md（修改）
② PATCH 已有的 umbrella skill 没有加载但有同类 skill 覆盖 ~/.hermes/skills/<name>/SKILL.md（修改）
③ ADD support file 该加 references/<topic>.md / templates/<name> / scripts/<name> 3 种子目录
④ CREATE NEW skill 上面都不合适，且是 class-level 命名 ~/.hermes/skills/<new-name>/SKILL.md（新建）
为什么"加载过"最重要？
因为加载过 = LLM 已经认为这个 skill 相关。往它身上加东西，比新建一个"理论上相关"的 skill 命中下次场景的概率高得多。Hermes 团队把这一点提升到了"first-class"——优先级 1 不需要 LLM 重新判断相关性。
§ 7

4 类信号 · 任意一个就动手

review agent 拿到对话后会扫这 4 类信号。任一触发就动手写 skill。

信号 1 · 用户纠正了风格/格式

"stop doing X"、"this is too verbose"、"don't format like this"、"why are you explaining"、"just give me the answer"、"you always do Y and I hate it"、或显式"remember this"。这是 FIRST-CLASS skill 信号，不只是 memory 信号。

为什么 style/format 进 skill 不进 memory：memory 是关于"你是谁、当前状态"，skill 是关于"怎么为这个用户做这类事"。风格偏好属于后者——同一类任务下次出现时，要按上次纠正的方式做。

信号 2 · 用户纠正了工作流

"no, first do X then Y, not Y then X"、"before running that, check Z"——编码为 SKILL.md 的 pitfall 或显式步骤。

信号 3 · 出现了非平凡的技术/修复

debug 路径、workaround、tool 用法、API quirk——任何未来 session 能受益的东西。捕获。

信号 4 · 已加载的 skill 居然是错的

最重要的一条。如果一个 skill 已经被这次对话加载/调过，但发现它过时/缺步骤/错了——立刻 patch 它，不要等下次。

§ 8

完整案例 · 机械臂 PID 抖动

拿你的 sagittarius_class 课里"调 PID"环节做一个完整走查。

对话上下文

学生
机械臂 joint1 一直抖动，ros2 topic 看 position 在 0.001 范围里高频抖。
agent
我查一下控制器配置……[调 terminal 跑 ros2 param list / ros2 control list_controllers]……
找到了：joint1 的 PID 控制器 P=0.1。对负载大的关节太低了。建议改 P=1.0。
学生
改成 1.0 试试……好了，不抖了。
agent
✅ 稳了。我把这个经验沉淀到 skill 里，下次你再遇到类似问题不用重新发明。

触发 1 跑完一轮后

学生说完"好了" → turn_finalizer.py:376 判断 → _should_review_skills = True → 起 daemon 线程 fork AIAgent → review agent 读完整对话 → 评估 4 类信号 → 信号 3 命中（非平凡修复）

review agent 写盘的 4 选 1 决策

按优先级：

  1. 对话里调过 holo-robotics-ros2-bridge 吗？——是（学生用了"ros2 topic"命令）
  2. 走优先级 ①：PATCH 这个 skill
  3. 在 SKILL.md 的 "Pitfalls" 段加一条："joint 抖动 99% 是 PID P 参数太低；sagittarius 关节 1/2/3 经验值 P≥1.0"

写盘后的实际产物

# 写到哪里：~/.hermes/skills/holo-robotics-ros2-bridge/SKILL.md

## Pitfalls

### joint 持续高频抖动
- 99% 是 PID 控制器 P 参数太低
- 检查：ros2 control list_controllers + ros2 param get /sgr532/joint1_position_controller pid.p
- 经验值：sagittarius 关节 1/2/3 需 P >= 1.0；关节 4/5/6 需 P >= 0.5
- 改了之后用 ros2 topic echo 看 position，标准差 < 0.001 算稳

下次类似场景

另一个学生问"机械臂抖动"，LLM 加载 holo-robotics-ros2-bridge，SKILL.md 里已经有这条 pitfall —— 第一次回答就提到"检查 PID P 参数"，不用重新发明。

这就是"持续学习"的物理实现：每轮对话可能产生一段新的经验，被自动写到 SKILL.md。下次同类问题出现，LLM 已经知道这个经验。这个循环跑得越久，skill 库越丰富，agent 越"懂你的领域"。
§ 9

3 道保险 · 防误伤

"自动写盘"听起来很危险——LLM 判断错了怎么办？Hermes 留了 3 道保险：

保险 机制 何时用
1 · 工具白名单 review agent 只能调 memory + skill_manage，其他工具运行时拒绝 始终（默认）
2 · write_approval 开关 skills.write_approval: true 时所有写盘都进入"staged"——写到 ~/.hermes/pending/skills/，要人审 小模型容易误判 / 安全环境 / 想要人审
3 · guard_agent_created 扫描 内容扫描器：检测危险模式（rm -rf /、写系统文件、暴露 secret） 始终（独立于 approval）

人审流程（write_approval 模式）

开启 write_approval: true 后，所有 skill 写盘都进 staging：

# 飞书 / CLI / Telegram 都用同一组命令
/skills pending                  # 列出所有待审 + 一行摘要
/skills diff <id>               # 看完整 diff
/skills approve <id>            # 通过 → 落盘
/skills reject <id>             # 拒绝 → 丢弃
/skills approval on              # 开/关 + 持久化
对教学场景的建议：前期建议开 write_approval: true，老师人工 review 每一份新生成的 skill。等模型行为稳定了，再关掉让它自动跑。
§ 10

对机器人 + 课程库的实操影响

1. 你不用手写 7 个 skill

Hermes 跑 1-2 周后，常用模式会被自动沉淀：

  • 学生问"ROS2 domain ID 怎么设" → 自动生成 holo-robotics-ros2-config skill
  • 学生问"joint 抖动" → 自动沉淀到 holo-robotics-ros2-bridge
  • 学生问"slam 怎么建图" → 自动沉淀到 holo-robotics-slam

2. 你应该写什么

只写"骨架 + 入口"：

  • 3-5 个 class-level umbrella skill（holo-robotics-ros2-bridge / holo-robotics-moveit / holo-robotics-slam / holo-edu-course-rag / holo-edu-tutor）
  • 每个的 SKILL.md 只写触发条件 + 主流程，不写细节——细节由触发 1 自动填
  • 你 override 或 pin（~/.hermes/skills/<name>/.pinned）的 skill curator 不会动

3. 错例库怎么搭

学生答错的题，被 Hermes 写到 memory 还是 SKILL.md？

  • memory：这个学生错了什么（个性化）
  • SKILL.md：这类学生容易错什么（通用化）

Hermes 会自动判断。如果一个错例 3 个学生都栽了，review agent 会把它从 memory 升级到 SKILL.md（加进对应课的"教学 pitfall"段）。

4. 时间线预期

阶段现象说明
第 1 周几乎没变化首次安装，curator 不会跑，触发 1 也才刚开始计数
第 2-4 周偶尔看到 💾 Patched holo-robotics-ros2-bridge触发 1 开始有动作，但很挑剔
第 5-8 周skill 库 7-15 个trigger 1 高频 + 第一次 curator 跑
第 9 周起开始有 stale / archivecurator 开始清理
3 个月后稳定的"越用越聪明"循环写、读、清理三件套都稳定
回到主阅读手册：读完附录 A + B，你应该能完全回答 Q1：Hermes 的"持续学习"对你意味着什么。
答案大概率是：值得。不是因为它的 LLM 强（我们用 DeepSeek），而是因为这套 沉淀 + 检索 + 清理 机制是工程化的、有代码保证的。
