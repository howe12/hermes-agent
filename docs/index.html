§ 1

"长"和"快"各自指什么

你问的是两件事：

  • "长期" = 数据能不能跨进程、跨月、跨年存活——不丢
  • "快速找到" = 在 10 万条记录里搜"机械臂挥手"能不能在 50ms 内返回——不快就废了

这两件事由两个不同技术解决：

长
SQLite 本地文件
数据写进 ~/.hermes/state.db 一个文件。进程退出、机器重启都不丢。WAL 模式保证崩溃也安全。
快
FTS5 倒排索引
"词 → 哪些行"反向建索引。查 "机械臂" 不用扫 10 万行，索引直接告诉你"在第 4, 17, 238, 901 行"。
准
BM25 排名
同样包含"机械臂"的 100 行，按"相关度"打分排前 10 个返回。

三者合起来，就是 Hermes 长期记忆的全部秘密。下面逐层拆。

§ 2

3 层架构 · 短 / 中 / 长期

Hermes 的"记忆"不是单一文件。它有 3 个不同时间尺度的组件，对应不同用途：

层存在形式寿命容量用途
短期 · 上下文 当前会话的 messages 单次对话 ~200K tokens（模型上限） 让 LLM 看到"刚才说了什么"
中期 · MEMORY.md Markdown 文件 永久（手动管理） 2,200 chars ≈ 800 tokens agent 自己写的"个人笔记"——项目、约定、用户偏好
长期 · SQLite + FTS5 SQLite 数据库 永久（自动累积） 无限（10万+ 条） 所有历史对话、所有工具调用、所有结果——可全文检索
关键点：中期和长期是两套机制。MEMORY.md 是 agent 主动维护的精炼笔记（agent 觉得"这条值得记"才写）；SQLite 是所有对话的自动存档（机器自己塞的，不删不挑）。前者进入 system prompt，后者通过 session_search 工具被检索。

当用户问"上次我们聊到哪了"时发生什么

这会触发 LLM 决定：调 session_search 工具。工具代码长这样（伪代码 + 真实查询骨架）：

# agent/skill_commands.py / run_agent.py 里的搜索调用
def search_history(query: str, limit: int = 10):
    safe_query = _sanitize_fts5_query(query)   # § 8 讲这个
    return session_db.search_sessions(safe_query, limit=limit)

# hermes_state.py 实际查询（行 3292）
sql = """
    SELECT
        m.id, m.session_id, m.role,
        snippet(messages_fts, 0, '>>>', '<<<', '...', 40) AS snippet,
        m.content, m.timestamp, m.tool_name, s.source, s.model,
        s.started_at AS session_started
    FROM messages_fts
    JOIN messages   m ON m.id = messages_fts.rowid
    JOIN sessions   s ON s.id = m.session_id
    WHERE messages_fts MATCH ?
    ORDER BY rank
    LIMIT ? OFFSET ?
"""

返回的结果里，snippet 是带 >>><<< 高亮关键词的摘录——LLM 看到这个就知道"啊上次确实讲过这个"。

§ 3

SQLite FTS5 是什么 · 跟普通 LIKE 比

很多人以为"全文搜索"就是 WHERE content LIKE '%机械臂%'。FTS5 比这个强 N 个数量级。

用 LIKE 搜索 10 万条记录的灾难

# 普通 LIKE 查询
SELECT * FROM messages WHERE content LIKE '%机械臂%';

# 发生了什么：
# 1. SQLite 打开 messages 表
# 2. 顺序扫描每一行（10 万行）
# 3. 每一行都做字符串包含判断
# 4. 把匹配的行收集起来
# → 耗时：O(N × M) · 10万行 × 每行 1000字符 = 1亿次字符比较
# → 实际：500ms - 5s（取决于机器）

FTS5 用倒排索引把这个变成 O(log N)

FTS5 在写数据时维护了一个反向索引：

# 文档原文（10 万条消息）：
[1] "今天演示机械臂挥手"
[2] "moveit 配置好以后开始规划"
[3] "学生问题：PID 控制器怎么调"
[... 99998 条]

# FTS5 自动建的倒排索引：
"机械臂"  →  [1, 17, 238, 901, ...]
"挥手"    →  [1, 45, 892]
"moveit"  →  [2, 56, 1024]
"pid"     →  [3, 67, 2003]

# 查 "机械臂"：直接 hash 查 → 拿到 5 个 rowid → 5 次精确定位
# 耗时：O(1) + 5 次 lookup · 实际 < 5ms

这不是魔法——是搜索引擎、数据库的通用底层技术。Elasticsearch、Lucene、PostgreSQL 的 GIN 索引、MySQL 的 FULLTEXT 索引，原理都一样。

为什么选 SQLite FTS5 而不是 Elasticsearch：
· 零运维：一个 .db 文件，git 可备份，dropbox 同步
· 进程内：不需要起第二个服务，没有网络开销
· 够用：FTS5 在 100 万行内都是 ms 级
· ACID：崩溃不丢数据，比 ES 强
· 你 NUC 8G 内存：SQLite 用 ~50MB，ES 用 ~2GB
§ 4

Hermes 怎么建表（带真实 DDL）

文件 hermes_state.py 行 602-625。这是真实代码，不是伪代码：

FTS_SQL = """
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts USING fts5(
    content
);

CREATE TRIGGER IF NOT EXISTS messages_fts_insert AFTER INSERT ON messages BEGIN
    INSERT INTO messages_fts(rowid, content) VALUES (
        new.id,
        COALESCE(new.content, '') || ' ' || COALESCE(new.tool_name, '') || ' ' || COALESCE(new.tool_calls, '')
    );
END;

CREATE TRIGGER IF NOT EXISTS messages_fts_delete AFTER DELETE ON messages BEGIN
    DELETE FROM messages_fts WHERE rowid = old.id;
END;

CREATE TRIGGER IF NOT EXISTS messages_fts_update AFTER UPDATE ON messages BEGIN
    DELETE FROM messages_fts WHERE rowid = old.id;
    INSERT INTO messages_fts(rowid, content) VALUES (
        new.id,
        COALESCE(new.content, '') || ' ' || COALESCE(new.tool_name, '') || ' ' || COALESCE(new.tool_calls, '')
    );
END;
"""

逐行解释

  • CREATE VIRTUAL TABLE ... USING fts5(content) —— 建一张虚拟表，不用你手写索引，FTS5 自动维护倒排索引
  • rowid —— 跟 messages 表的主键 id 保持一致（关键！见 § 5）
  • content —— 只存一列：你索引的文本
  • 3 个 TRIGGER —— 监听 messages 表的 INSERT/DELETE/UPDATE，自动同步到 FTS 表

被索引的字段

注意 INSERT 那行的拼接：

COALESCE(new.content, '') || ' ' || COALESCE(new.tool_name, '') || ' ' || COALESCE(new.tool_calls, '')

意思是：对话内容 + 工具名 + 工具调用参数 全部进索引。也就是说你搜"机械臂"的时候，不只匹配到对话里说过机械臂的句子，还会匹配到曾经调过"ros2 控制机械臂"那条工具调用。

这在调试时极有用——学生问"上次用的是什么命令"，搜索"ros2 topic"就能找到对应的 tool call。

§ 5

Trigger 怎么让记忆"自动同步"

上面的 3 个 TRIGGER 是整个机制最巧妙的地方。应用代码完全不需要知道 FTS 表的存在。

应用代码 → INSERT INTO messages → trigger 自动 → INSERT INTO messages_fts → 倒排索引更新

这就是为什么 Hermes "长期"——你从来没主动去写 FTS 表，数据库自己保证一致性。即使应用代码崩了，要么 messages 和 FTS 都写入，要么都没写入（ACID 事务）。

对比常见错误做法

很多 RAG 系统是这样写的：

# ❌ 错误做法（很多 RAG 系统这么干）
def save_message(text):
    db.execute("INSERT INTO messages VALUES (?)", (text,))    # step 1
    db.execute("INSERT INTO fts_index VALUES (?)", (text,))  # step 2
    # 如果 step 2 失败（程序崩了/连接断了），FTS 索引和原文不一致
    # 搜不到 → 长期记忆"漏"了

Hermes 用 trigger 的好处：

# ✓ Hermes 做法
def save_message(text):
    db.execute("INSERT INTO messages VALUES (?)", (text,))
    # SQLite 在同一个事务里自动：
    #   1. 插入 messages
    #   2. 触发 AFTER INSERT trigger
    #   3. trigger 把数据塞进 messages_fts
    # 要么都成功，要么都失败。原子性。
§ 6

查询为什么快 · BM25 排名

FTS5 不只是"找出包含 X 的行"，它会排序——最相关的在前。

排序算法叫 BM25（Best Matching 25）。原理不复杂，但很巧妙：

3 个评分维度

  1. 词频（TF）：匹配词在文档里出现越多，分数越高
  2. 逆文档频率（IDF）：如果"机械臂"在 100 万行里只出现 5 次（很罕见），它比"了"（出现 50 万次）的权重高得多
  3. 文档长度归一化：短文档比长文档更"专注"——同样 1 个匹配，10 字的短句比 1000 字的长文相关性高

这三者合一，"机械臂挥手" 会自动排到只提一句机械臂的段落前面，而不是排在学生夸夸其谈 PID 的长文里。

FTS5 的查询语法

Hermes 的实际查询用 MATCH 操作符：

-- 简单：包含任一词
WHERE messages_fts MATCH '机械臂 OR 挥手'

-- 短语：必须连在一起
WHERE messages_fts MATCH '"机械臂 挥手"'

-- 必含 + 排除
WHERE messages_fts MATCH '机械臂 AND moveit NOT gazebo'

-- 字段限定
WHERE messages_fts MATCH 'content:机械臂'

-- 排名（BM25 自动算）
ORDER BY rank
LIMIT 10

rank 这个伪列是 FTS5 自动提供的——倒排索引里 BM25 分高的行排前面。

§ 7

中文怎么办 · trigram 分词器

这一节是 Hermes 对我们项目最关键的部分——因为你的 7 门课几乎全是中文。

问题：默认 FTS5 对中文不友好

FTS5 默认用 unicode61 分词器，规则很简单：按 Unicode 类别切词。但中文每个字都是独立 token：

"机械臂挥手"
# unicode61 分词：
["机", "械", "臂", "挥", "手"]

# 搜 "机械臂"：变成 "机 AND 械 AND 臂" → 必须 3 个字同时出现
# 搜 "机械臂挥手"：变成 5 个字 AND → 几乎搜不到完整短语

解决：trigram 分词器

Hermes 用了第二张FTS5 虚拟表，专门给中文/子串搜索用（hermes_state.py 行 626）：

FTS_TRIGRAM_SQL = """
CREATE VIRTUAL TABLE IF NOT EXISTS messages_fts_trigram USING fts5(
    content,
    tokenize='trigram'
);
"""

trigram 把文本切成3 字节滑动窗口：

"机械臂挥手" (UTF-8: 15 字节)
# trigram 分词：
["机械臂", "械臂挥", "臂挥手"]

# 搜 "机械臂"：直接命中第一个 trigram ✓
# 搜 "械臂挥"：也能命中（子串搜索） ✓
# 搜 "机械臂挥手"：需 ≥9 字节 → 3 个 CJK 字 → 正好够 ✓

代码里怎么选择走哪张表

行 3373 的判断逻辑：

is_cjk = self._contains_cjk(query)
if is_cjk:
    cjk_count = self._count_cjk(raw_query)
    if cjk_count >= 3 and not _any_short_cjk:
        # 走 trigram 表（精确短语 + BM25）
        # WHERE messages_fts_trigram MATCH '"机械臂"' AND '"挥手"'
    else:
        # 中文 ≤2 字，trigram 拼不出来 → 降级用 LIKE
        # WHERE content LIKE '%机%' AND content LIKE '%械%' ...

也就是：

  • 英文 / 数字 / 混合查询 → 走 messages_fts（unicode61）
  • 中文 ≥3 字 → 走 messages_fts_trigram
  • 中文 ≤2 字 → 降级 LIKE（慢但能用）
对我们的意义：你的 7 门课如果是中文，搜"机械臂挥手"在 Hermes 里能精准命中，不会拆字失败。trigram 索引体积大 ~2 倍（多了重叠的 3-gram），但 7 门课总量也就几 MB，磁盘无压力。
§ 8

查询安全 · sanitize

这是个容易踩坑的点。FTS5 的 MATCH 有自己的语法（带 "()+-*:^ 这些特殊字符），如果直接拼用户输入，SQLite 会抛 OperationalError。

Hermes 用 _sanitize_fts5_query 函数（行 3138）清洗输入。简化版：

@staticmethod
def _sanitize_fts5_query(query: str) -> str:
    # Step 1: 保留双引号包裹的精确短语
    #   "moveit 配置" → 保留不动
    #   单独一个 " → 标记成占位符
    def _preserve_quoted(m): _quoted_parts.append(m.group()); return f"\\x00Q{len(_quoted_parts)-1}\\x00"
    sanitized = re.sub(r'"[^"]*"', _preserve_quoted, query)

    # Step 2: 去掉剩下的 FTS5 特殊字符
    sanitized = re.sub(r'[+{}():"^]', " ", sanitized)

    # Step 3: 含连字符/点号的项加引号（避免被切分）
    #   chat-send → "chat-send"（不切成 "chat" + "send"）
    #   P2.2 → "P2.2"（不切成 "P2" + "2"）
    sanitized = re.sub(r'[A-Za-z0-9_]+[-.][A-Za-z0-9_.-]*',
                     lambda m: f'"{m.group()}"', sanitized)

    # Step 4: 把占位符还原
    return "".join(...)

为什么要做这个？

  • 学生输入 TODO: fix bug → 冒号 : 是 FTS5 的"列限定符"，会触发 "no such column" 错误
  • 学生输入 (2024 笔记) → 括号被当成语法
  • 学生输入 moveit + 配置 → 加号被当成 AND 强制

sanitize 把这些都转成安全形式，不破坏正常搜索，避免 SQL 错误。

§ 9

一次完整写入 + 检索的全流程

把所有环节串起来。场景：学生问"上次我问你机械臂怎么挥手来着"。

阶段 1 · 写入（昨天的对话）

学生发问 → agent 回答 → save_turn(user_msg, assistant_msg) → INSERT INTO messages → trigger 触发 → messages_fts 自动更新

这一步 agent 跑完后，数据库里永久存着。你明天重启电脑，后天刷系统，文件还在。

阶段 2 · 检索（今天）

学生发问 → LLM 决定调 session_search → query = "机械臂 挥手" → _sanitize_fts5_query → MATCH '"机械臂" "挥手"' → 命中 N 行 · BM25 排序 → top-10 + snippet 高亮 → 塞进 LLM 上下文 → LLM 写回答

整个流程总耗时

  • 写入：~2ms / 条（含 trigger 触发 + 索引更新）
  • 检索：~5-20ms（10 万行规模下）
  • snippet 生成：~1ms
  • 总延迟：~30ms —— 用户感觉"瞬间"
§ 10

对我们的项目意味着什么

读完这一篇，你对 aquarius-psyche 项目能做的判断是：

1. 7 门课接进 Hermes 的可行性 ✅

7 门课总量预估几十 MB。每节课拆成 100-500 个 chunk，7 × 300 ≈ 2100 chunk。
塞进 messages_fts 索引，~10MB 索引文件，每次查询 5-20ms。
够用，不卡。

2. 错例库能撑多久 ✅

假设一个学生每周产生 50 条错例，一年 50 周 = 2,500 条/年/学生。
一个班 30 个学生 = 75,000 条/年/班。
3 年下来 ~22 万条。FTS5 仍能 30ms 内返回，磁盘 < ~200MB。
够用 5-10 年不爆。

3. 跟 ChromaDB 怎么分工

Hermes 自带 FTS5 处理对话历史，不需要ChromaDB。
但是——7 门课是静态知识，不是对话历史。两种方案：

  • 方案 A（轻）：把 7 门课当"超大 MEMORY.md"塞进 FTS5 索引。简单、零依赖。
  • 方案 B（重）：用 ChromaDB 单独索引 7 门课，学嵌入向量。能搜"意思相近"的内容，不只"包含关键词"。

建议：先 A 跑通，再 B。A 用 Hermes 自带能力，不增加任何依赖，就能让你看到 80% 的效果。

4. 还能扩展什么

  • 检索结果再排序：用 LLM 对 top-10 重新打分（re-rank），精度再提升 20%
  • 按时间衰减权重：1 年前的对话权重 ×0.5，让最近的优先
  • 多模态索引：图片转 caption 后进 FTS5（已有 vision tool）
  • 导出训练数据：把 FTS5 命中的对话导出成 ShareGPT 格式，微调你自己的小模型（这是 Hermes 的"研究就绪"卖点之一）
